# Obsidian 知识库启动器开发计划

## 1. 计划目标

依据《Obsidian 知识库一条龙启动器设计方案》，完成一个面向 Windows 10/11 的 MVP：普通用户无需理解 Git，即可安装、打开、更新和修复一个由公开 GitHub Releases 发布的 Obsidian 知识库。

本计划优先验证完整技术链路，再开发正式界面和更新回滚能力。计划以 1 名全职开发者为基准，预计 6 周完成首个可发布测试版；若由多人并行开发，可按模块缩短周期，但阶段验收顺序不变。

## 2. MVP 完成标准

满足以下条件时，MVP 才视为完成：

- 可在全新 Windows 10/11 用户环境中运行自包含启动器；
- 能识别 Obsidian 是否安装及其版本是否满足要求；
- 未安装或版本过低时，能明确引导用户通过许可范围内的安装来源完成安装；
- 能从指定公开 GitHub Release 下载知识库、清单和校验文件；
- 下载内容必须先进入 `staging`，通过路径安全检查和 SHA-256 校验后才能进入正式目录；
- 能按锁定版本安装插件，并保留插件 `data.json`；
- 能引导首次注册 Vault，并在后续运行时打开清单指定首页；
- 能检查并应用新版本，保留规定的本地状态；
- 更新失败时能恢复到上一个可用版本；
- 能执行修复安装并产出可定位问题的日志；
- 核心自动化测试、干净环境端到端测试和发布检查全部通过。

## 3. 开发原则与约束

- 正式 Vault 不接受未经验证的文件；下载、解压和验证均在 `staging` 中完成。
- 所有文件操作必须防止 ZIP Slip、非法相对路径、符号链接越界和意外删除工作目录之外的文件。
- 更新正式目录采用可恢复的目录切换策略，不能直接逐文件覆盖后再尝试补救。
- 网络调用统一支持超时、取消、有限重试和清晰错误分类。
- 版本比较使用语义化版本规则或项目明确规定的版本规则，不使用普通字符串比较。
- 日志不得记录令牌、完整用户隐私路径或其他敏感信息。
- 第一版不实现私有仓库、多知识库、用户内容合并、后台常驻和跨平台支持。

## 4. 开始开发前必须确认的输入

以下内容不阻塞项目骨架搭建，但必须在进入真实联调前确定：

| 输入 | 要求 | 最晚确认阶段 |
|---|---|---|
| GitHub 仓库 | 明确 `owner/repo`、Release 资产命名和 API 访问方式 | 阶段 1 |
| Vault 标识 | 固定 `id`、显示名称、入口笔记路径 | 阶段 1 |
| Obsidian 来源 | 明确随包分发、WinGet 或官方下载页的优先级和许可边界 | 阶段 1 |
| 最低版本 | 明确最低 Obsidian 版本及测试版本 | 阶段 1 |
| 插件清单 | 列出插件 ID、仓库、锁定版本、文件哈希和许可证结论 | 阶段 2 |
| 更新策略 | 审核 `overwrite`、`preserve`、`merge` 的真实文件范围 | 阶段 3 |
| 应用身份 | 产品名、图标、发布者、版本规则和代码签名方案 | 阶段 4 |

## 6. 详细任务拆解

### 阶段 0：工程基础

#### 0.1 解决方案与项目结构

- 创建 .NET 10 WPF 解决方案；
- 建议按职责拆分为：
  - `KnowledgeLauncher.App`：WPF 界面、应用启动和依赖装配；
  - `KnowledgeLauncher.Core`：领域模型、接口、清单和更新规则；
  - `KnowledgeLauncher.Infrastructure`：网络、文件系统、进程、注册表和日志实现；
  - `KnowledgeLauncher.Tests`：单元测试；
  - `KnowledgeLauncher.IntegrationTests`：文件、HTTP 和更新集成测试；
- 启用 nullable、警告检查、统一代码格式和集中包版本管理；
- 建立 Debug、Release 和自包含单文件发布配置。

验收标准：

- 本地一条命令可完成还原、构建和测试；
- Release 构建可生成 Windows x64 自包含产物；
- UI 项目不直接实现下载、解压或更新业务逻辑。

#### 0.2 配置、目录与日志

- 定义 `launcher.json` 及其版本字段；
- 实现 `%LOCALAPPDATA%\KnowledgeLauncher` 目录解析；
- 实现 `app`、`vault`、`staging`、`backup`、`cache`、`logs` 初始化；
- 实现结构化文件日志、日志轮转和用户可读错误编号；
- 抽象时钟、文件系统和 HTTP 客户端，便于测试。

验收标准：

- 首次启动能自动创建目录和默认配置；
- 配置损坏时不会崩溃，可备份损坏文件并恢复默认值；
- 日志写入失败不会阻断主流程。

### 阶段 1：最小技术验证

#### 1.1 Obsidian 检测与启动

- 按优先级检测注册表、常见安装路径和 URI 协议；
- 读取并解析已安装版本；
- 对比清单要求的最低版本；
- 实现 `obsidian://open` URI 构造、参数编码和启动；
- 对首次尚未注册的 Vault 显示明确引导。

验收标准：

- 已安装、未安装、版本过低、URI 未注册四种情况均能正确识别；
- Vault 名称、路径和入口文件含中文或空格时仍能正确打开；
- 进程启动失败时显示可执行的恢复建议。

#### 1.2 Release 查询与下载

- 实现 GitHub Releases `latest` 查询；
- 按确定的命名规则选择 ZIP、`manifest.json` 和 `checksums.json`；
- 实现流式下载、临时文件、进度上报、取消、超时和有限重试；
- 设置明确的 `User-Agent`，处理限流、404、网络中断和资产缺失。

验收标准：

- 不把整个大文件一次性载入内存；
- 中断下载不会留下被误认为完整的正式文件；
- 缺少必需资产时停止安装并给出明确错误。

#### 1.3 安全解压与清单验证

- 定义并解析 `manifest.json`、`plugin-lock.json`、`update-policy.json` 和校验清单模型；
- 拒绝未知或不支持的清单结构版本；
- 校验资产 SHA-256；
- 解压前检查目标路径，阻止 `../`、绝对路径、盘符路径和目录越界；
- 解压后验证必需文件、入口文件和清单内部引用。

验收标准：

- 被篡改资产、错误哈希、恶意 ZIP 路径和缺失入口文件全部被拒绝；
- 验证失败时正式 Vault 不发生变化；
- 错误日志能指出失败资产和校验阶段。

#### 1.4 单插件锁定安装原型

- 读取一个真实插件的锁定记录；
- 从插件官方 GitHub Release 下载锁定版本文件；
- 校验 `main.js`、`manifest.json` 和可选 `styles.css`；
- 安装到 `.obsidian/plugins/{plugin-id}`；
- 更新 `.obsidian/community-plugins.json`；
- 插件最低 Obsidian 版本不满足时停止流程。

阶段验收演示：

```text
检测 Obsidian
→ 下载测试知识库 Release
→ 校验并安全解压
→ 安装一个锁定版本插件
→ 引导注册 Vault
→ 使用 Obsidian URI 打开首页
```

只有上述链路在至少两台环境不同的 Windows 机器或虚拟机上稳定通过，才能进入正式首次安装开发。

### 阶段 2：首次安装 MVP

#### 2.1 安装流程编排

- 将检测、下载、验证、插件配置和正式目录切换编排为可恢复的状态机；
- 为每一步定义状态、进度、用户文案、可重试性和错误码；
- 应用退出后再次启动时，能识别并清理或继续处理遗留 `staging`；
- 安装成功后持久化 Vault ID、本地版本、安装时间和最近成功状态。

#### 2.2 多插件安装

- 支持多个插件顺序或有限并发下载；
- 精确锁定插件版本，不自动追踪最新版；
- 统一处理插件无 `styles.css` 的情况；
- 生成稳定、去重的 `community-plugins.json`；
- 首次运行时展示社区插件代码信任提示，并记录用户确认。

#### 2.3 Obsidian 安装引导

- 根据最终许可与产品决策实现安装来源策略；
- 若使用 WinGet，先检测 WinGet 可用性并展示确认步骤；
- 若 WinGet 不可用或失败，提供官方下载页入口；
- 安装返回后重新检测，而不是假定安装成功。

验收标准：

- 用户取消安装时安全退出且不残留半成品 Vault；
- 安装完成后无需重启启动器即可重新检测；
- 所有界面文案避免出现 clone、pull、commit、分支等概念。

### 阶段 3：更新、回滚与修复

#### 3.1 版本检查

- 保存本地已安装版本并与远程清单版本比较；
- 支持“已是最新”“有更新”“远程版本不可用”“本地状态未知”；
- 启动时检查和用户手动检查共用同一服务；
- GitHub 暂时不可用时允许用户打开现有可用 Vault。

#### 3.2 更新策略执行器

- 实现 `overwrite`、`preserve` 和 `merge` 规则解析；
- 明确定义规则优先级、路径匹配语义和冲突行为；
- 为 `.obsidian/app.json`、`.obsidian/appearance.json` 实现字段级 JSON 合并；
- 默认保留 workspace、hotkeys 和插件 `data.json`；
- 在切换前检查 Obsidian 是否正在使用该 Vault，并请求用户关闭。

验收标准：

- 每条策略均有单元测试和包含中文路径的集成测试；
- 同一路径命中多个规则时结果确定且有日志；
- 无法安全合并或目标文件 JSON 损坏时停止更新，不猜测覆盖。

#### 3.3 原子切换与回滚

- 新版本完整构建并验证后，才进入目录切换；
- 切换前把当前版本保存为唯一一个 `backup`；
- 切换失败时恢复原 Vault；
- 更新成功并再次验证后再清理旧临时文件；
- 对磁盘空间不足、文件占用、断网、应用被终止和哈希失败进行故障注入测试。

验收标准：

- 任一预设故障发生后，系统最终只处于“旧版可用”或“新版可用”状态；
- 不出现新旧文件混合的正式 Vault；
- 回滚后本地版本信息与实际 Vault 一致。

#### 3.4 修复安装

- 扫描知识库和插件程序文件完整性；
- 重新下载缺失或损坏的受管理文件；
- 保留所有策略声明的本地状态文件；
- 修复前显示影响范围，修复后生成结果摘要。

### 阶段 4：正式界面与产品化

#### 4.1 主窗口

- 展示知识库名称、当前版本、Obsidian 状态、插件状态和上次更新时间；
- 提供“打开知识库”“检查更新”“修复安装”三个主操作；
- 正在执行独占任务时禁用冲突操作；
- 长任务支持进度、取消和关闭确认。

#### 4.2 首次运行与错误体验

- 首次运行使用分步骤进度页；
- 对网络、权限、磁盘、Obsidian、校验和文件占用错误提供不同文案；
- 错误页提供“重试”“打开日志目录”或“继续打开现有版本”等适用操作；
- 明确提示上游笔记更新可能覆盖用户直接修改。

#### 4.3 可访问性与本地化基础

- 确保键盘可操作、焦点顺序合理、控件具有辅助名称；
- 在 100%、150%、200% 缩放下检查布局；
- 用户文案从代码中分离，为后续多语言保留资源结构。

### 阶段 5：发布流水线与验收

#### 5.1 知识库发布流水线

- 建立 GitHub Actions：过滤私有文件、验证清单、生成版本、打包 ZIP、生成 SHA-256、创建 Release 和上传资产；
- 发布前检查入口文件、插件锁定记录、重复路径和敏感文件规则；
- 固定 Action 版本或提交 SHA，最小化工作流权限；
- 保留可复现的发布资产命名规范。

#### 5.2 启动器发布

- 生成 Windows x64 自包含单文件版本；
- 填写文件版本、产品版本、图标和发布者信息；
- 接入代码签名并验证签名；
- 生成 SHA-256 和发布说明；
- 在干净环境验证不依赖开发机 SDK 或运行时。

#### 5.3 最终测试矩阵

| 类别 | 必测场景 |
|---|---|
| 系统 | Windows 10、Windows 11；标准用户；中文用户名路径 |
| 显示 | 100%、150%、200% 缩放 |
| Obsidian | 未安装、符合最低版本、版本过低、URI 未注册 |
| 网络 | 正常、超时、断网、下载中断、GitHub 限流、资产 404 |
| 文件 | 磁盘空间不足、文件占用、只读、损坏 ZIP、错误哈希、恶意路径 |
| Vault | 首次安装、已有最新版、跨版本更新、更新失败回滚、修复安装 |
| 插件 | 无样式文件、多插件、哈希错误、版本不兼容、保留 `data.json` |
| 路径 | 空格、中文、长路径、入口文件重命名或缺失 |

发布门槛：

- 所有 P0/P1 缺陷关闭；
- 核心服务单元测试全部通过；
- 首次安装、更新、回滚、修复四条端到端用例全部通过；
- 至少一次从真实 GitHub Release 到真实 Obsidian 打开的完整演练通过；
- 安装包或单文件产物在全新 Windows 虚拟机中验证通过；
- 日志、许可说明、社区插件提示和用户修改覆盖提示齐全。

## 7. 建议任务优先级

### P0：没有它就不能发布

- 安全下载、哈希验证和安全解压；
- Obsidian 检测、版本检查和打开 Vault；
- 清单解析及插件锁定安装；
- `staging`、正式目录切换、备份和回滚；
- 首次安装、检查更新和修复安装主流程；
- 错误日志和干净环境端到端测试。

### P1：测试版必须具备

- 正式 WPF 主窗口和进度体验；
- 下载取消与有限重试；
- 本地状态保留及 JSON 合并；
- 社区插件信任提示；
- 代码签名、发布说明和完整测试矩阵。

### P2：MVP 后再评估

- 数字签名清单 `manifest.sig`；
- 增量更新和断点续传；
- 多架构或 ARM64；
- 多知识库、私有仓库和用户内容合并；
- 启动器自身自动更新。

其中，发布资产 SHA-256 只能证明下载结果与发布清单一致；若需要抵抗发布清单或托管账号被篡改，应把 `manifest.sig` 提升到 P0/P1，并将公钥内置于启动器。

## 8. 关键风险与应对

| 风险 | 影响 | 应对 |
|---|---|---|
| Obsidian 分发许可或安装来源不明确 | 阻塞正式分发 | 阶段 1 前冻结安装来源策略并保留可替换接口 |
| 插件 Release 资产命名不一致 | 自动安装失败 | 锁定文件显式声明资产或下载 URL，不依赖统一猜测 |
| GitHub API 限流或不可用 | 无法检查更新 | 缓存最近成功结果，允许离线打开现有 Vault |
| 用户在 Obsidian 中编辑上游笔记 | 更新时内容丢失 | 更新前提示，严格按策略保留个人状态，MVP 不承诺合并 |
| Obsidian 占用文件 | 更新产生混合状态 | 更新前检测并要求关闭，目录级切换，失败即回滚 |
| 路径匹配规则理解不一致 | 覆盖错误文件 | 固定 glob 语义和优先级，使用真实 Vault 快照做回归测试 |
| SHA-256 清单与资产同源被替换 | 无法验证发布者身份 | 后续加入离线私钥签名的 `manifest.sig` |
| 单文件发布或裁剪破坏反射序列化 | 运行时失败 | 发布配置进入 CI，并在自包含产物上执行冒烟测试 |

## 9. 每周检查点

### 第 1 周

- 工程、配置、目录、日志和 CI 可用；
- Obsidian 检测与 URI 原型完成；
- 冻结测试仓库、资产命名和安装来源方案。

### 第 2 周

- Release 下载、安全解压、哈希验证完成；
- 单插件安装完成；
- 最小端到端技术验证通过。

### 第 3 周

- 首次安装状态机、多插件和安装引导完成；
- 干净 Windows 环境首次安装通过。

### 第 4 周

- 更新策略、备份、原子切换、回滚和修复完成；
- 故障注入测试通过。

### 第 5 周

- 正式 WPF 主窗口、进度、错误处理和可访问性完成；
- 核心流程冻结，只处理缺陷。

### 第 6 周

- GitHub Actions、签名、发布产物和说明完成；
- 完整测试矩阵通过，发布首个测试版。

## 10. 建议的首批开发任务

项目启动后按以下顺序领取任务：

1. 创建解决方案、测试项目和 CI 构建。
2. 定义配置模型、清单模型、错误模型和核心接口。
3. 实现应用数据目录与日志。
4. 实现 Obsidian 检测、版本比较和 URI 构造。
5. 建立仅含一个插件的测试 Vault Release。
6. 实现 Release 查询、资产选择和流式下载。
7. 实现 SHA-256 校验与安全 ZIP 解压。
8. 实现单插件锁定安装和 `community-plugins.json` 更新。
9. 编写并执行最小端到端技术验证脚本。
10. 根据验证结果评审架构，再进入正式首次安装状态机开发。
